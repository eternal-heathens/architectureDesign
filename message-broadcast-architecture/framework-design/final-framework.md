# 三角色技术架构输出框架

## 1. 框架定位

该框架用于把一个技术问题拆成多角色协作流程，并输出正式技术架构方案：

```text
用户问题
  -> 技术总监定义边界和评价标准
  -> 架构师设计总体方案
  -> 技术骨干补充详细设计并提出质疑
  -> 架构师修订方案
  -> 技术总监最终裁决和定稿
```

核心价值：

- 技术总监负责收敛和裁决。
- 架构师负责构建整体架构。
- 技术骨干负责落地细化和反向质疑。
- 输入材料过大时，先压缩成结构化技术问题摘要，再进入三角色架构评审链路。
- 遇到通用技术问题时，先调研成熟开源工具和行业方案，再结合业务目标、稳定性、成本、安全合规和团队能力综合决策。

## 2. 角色配置

| 角色 | 模型/工具 | 职责 |
|---|---|---|
| 技术总监 | Codex + GPT-5.5 | 接收问题、定义边界、要求开源与替代方案调研、提出质疑、最终裁决 |
| 架构师 | Codex + GPT-5.5 | 设计总体架构、模块边界、技术选型、开源工具评估和演进路线 |
| 技术骨干 | Claude Code + CC Switch / Codex gpt-5.5 | 详细设计、工程落地分析、开源工具可落地性验证、风险识别、质疑架构师方案 |

## 3. 技术选型原则

架构输出必须遵循“开源优先、验证先行、综合决策”的原则。

当问题属于通用技术能力、基础设施能力、工程效率能力或常见中间件能力时，不直接默认自研，应先调研是否已有成熟开源工具、框架或行业通用方案可用。

开源方案调研至少覆盖：

- 功能匹配度：是否覆盖当前核心需求和关键边界场景。
- 成熟度：社区活跃度、维护频率、版本稳定性和生产案例。
- 集成成本：与当前语言、框架、部署、监控和运维体系的适配难度。
- 性能与稳定性：吞吐、延迟、资源占用、故障恢复和高可用能力。
- 安全合规：License、漏洞记录、依赖风险和数据安全边界。
- 可扩展性：后续业务增长、二次开发和替换迁移能力。
- 团队掌握度：团队学习、排障、维护和持续演进成本。

最终决策不以“是否开源”作为唯一标准：

- 成熟开源工具能满足需求时，优先采用开源方案。
- 开源工具能覆盖主体能力但需要业务适配时，优先考虑基于开源工具扩展。
- 开源工具不成熟、维护风险高、License 不合规或与核心业务强耦合时，可以选择自研。
- 商业产品在稳定性、支持能力和交付周期上明显更优时，也应纳入候选方案。
- 涉及核心竞争力、强业务差异化或高安全要求时，必须明确自研理由和长期维护成本。

关键技术选型需要在最终方案中记录决策依据，包括候选方案、开源调研结论、PoC 结果、最终选择、取舍原因、风险和后续演进计划。

## 4. 输入压缩规则

当用户输入、历史方案、代码背景、日志、会议纪要或上下游约束材料过大时，必须先进行结构化压缩，再进入技术总监、架构师和技术骨干链路。

压缩目标不是简单缩短文字，而是保留架构决策必需信息：

- 背景、目标、非目标和成功标准。
- 当前系统现状、上下游边界和关键依赖。
- 已有技术方案、已确认结论、争议点和替代方案。
- 性能、成本、安全、合规、可用性、扩展性和交付周期约束。
- 数据规模、流量峰值、SLA、部署环境和运维要求。
- 风险、历史故障、异常场景和未决问题。
- 需要优先调研的通用技术问题和可能的开源工具方向。

压缩结果建议使用：

```markdown
## 输入压缩摘要
## 背景与目标
## 系统现状与边界
## 关键约束
## 已有方案与结论
## 争议点与未决问题
## 数据规模与性能目标
## 风险与异常场景
## 开源工具调研方向
```

后续角色只能基于压缩摘要继续推演；如果发现摘要缺少关键判断依据，应先补问或回看原始材料中的相关片段。

## 5. 工作流

```text
0. 输入过大时：生成结构化输入压缩摘要
1. 技术总监：问题理解与任务分派
2. 架构师：总体架构初稿
3. 技术骨干：详细设计与反向质疑
4. 架构师：基于质疑修订方案
5. 技术总监：最终评审与正式方案
```

## 7. 角色失败重试策略

- 单个角色超时或调用失败时，默认简单重试 1 次。
- 重试间隔默认 3 秒。
- 重试仍失败时，保留已完成阶段产物，并在 `run.json` 记录 `attempts`、`retries`、`retryErrors`。
- 配置错误类问题不反复重试，例如 CC Switch provider 不匹配、模型凭据缺失、命令配置缺失。
