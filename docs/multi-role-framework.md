# 多角色方案输出框架

本文件汇总当前三类多角色方案输出框架：产品方案输出、技术架构输出、详细设计文档输出。三个框架当前保持独立使用，后续可以按 `产品方案输出 -> 技术架构输出 -> 详细设计文档输出` 串联。

## 1. 触发词约定

以后在 Codex 中直接使用以下前缀即可：

```text
产品方案输出：
架构评审：
详细设计文档输出：
```

别名映射：

| 输入前缀 | 归属框架 |
|---|---|
| `需求方案输出：` | 产品方案输出 |
| `产品需求评审：` | 产品方案输出 |
| `技术架构输出：` | 技术架构输出 |
| `技术方案输出：` | 技术架构输出 |
| `架构方案输出：` | 技术架构输出 |
| `详细设计：` | 详细设计文档输出 |
| `研发详细设计：` | 详细设计文档输出 |

## 2. 统一角色失败重试策略

三个多角色框架统一采用简单重试策略：

```json
{
  "retry": {
    "maxRetries": 1,
    "backoffMs": 3000
  }
}
```

规则：

- 单个角色超时或调用失败时，默认简单重试 1 次。
- 重试间隔默认 3 秒。
- 重试仍失败时，保留已完成阶段产物，并标记失败角色和失败原因。
- 技术架构输出的可执行编排会在 `run.json` 中记录 `attempts`、`retries`、`retryErrors`。
- 配置错误类问题不反复重试，例如模型凭据缺失、provider 不匹配、命令不存在、baseUrl 缺失。

## 3. 产品方案输出

### 3.1 框架定位

用于把一个初始想法、业务问题或模糊需求，整理成可确认、可评审、可验收的正式产品方案文档。

流程：

```text
用户想法
  -> CEO 对话澄清与需求分析
  -> 产品总监产品化为 PRD
  -> 技术总监核对技术可行性、成本和风险
  -> 产品总监修订 PRD
  -> CEO 最终确认需求边界
  -> 输出最终产品方案 final-product-plan.md
```

### 3.2 角色配置

| 角色 | 建议模型/工具 | 职责 |
|---|---|---|
| CEO | Codex / GPT-5.5 | 和用户对话、理解业务目标、澄清场景、确认优先级、做最终需求裁决 |
| 产品总监 | Codex / GPT-5.5 | 生成产品化需求文档、用户故事、功能边界、验收标准、版本规划 |
| 技术总监 | Codex / GPT-5.5 或本地技术评审通道 | 核对技术可行性、成本、复杂度、依赖、风险和交付周期 |

### 3.3 阶段输出

```text
01-ceo-requirement-analysis.md
02-product-director-prd-draft.md
03-technical-director-feasibility.md
04-product-director-prd-revision.md
05-final-product-plan.md
```

最终产品方案应包含：需求结论、背景与目标、目标用户、核心场景、业务流程、功能范围、页面与交互要求、数据与权限要求、非功能需求、验收标准、版本规划、技术约束与成本结论、不做范围、未决问题。

### 3.4 推荐输入模板

```text
产品方案输出：
【背景】...
【目标用户】...
【核心目标】...
【当前问题】...
【约束】...
【希望输出】最终产品方案
```

## 4. 技术架构输出

### 4.1 框架定位

用于把一个技术问题拆成多角色协作流程，并输出正式技术架构方案。

流程：

```text
用户问题
  -> 技术总监定义边界和评价标准
  -> 架构师设计总体方案
  -> 技术骨干补充详细设计并提出质疑
  -> 架构师修订方案
  -> 技术总监最终裁决和定稿
```

### 4.2 角色配置

| 角色 | 模型/工具 | 职责 |
|---|---|---|
| 技术总监 | Codex + GPT-5.5 | 接收问题、定义边界、提出质疑、最终裁决 |
| 架构师 | Codex + GPT-5.5 | 设计总体架构、模块边界、技术选型和演进路线 |
| 技术骨干 | Claude Code + CC Switch / Codex gpt-5.5 | 详细设计、工程落地分析、风险识别、质疑架构师方案 |

### 4.3 阶段输出

```text
00-question.md
01-technical-director-intake.md
02-architect-draft.md
03-senior-engineer-review.md
04-architect-revision.md
05-final-architecture.md
run.json
```

最终技术架构方案应包含：背景与目标、需求理解、架构原则、总体架构、核心模块设计、关键业务流程、数据架构、接口与集成设计、技术选型、部署架构、安全设计、性能与扩展性设计、高可用与容灾设计、可观测性设计、工程实施计划、风险清单与应对策略、架构争议与取舍、最终结论。

### 4.4 推荐输入模板

```text
架构评审：
【背景】...
【目标】...
【约束】...
【关注点】高并发、可靠性、成本、扩展性、故障恢复
```

当前本机可执行命令：

```bash
npm run arch -- --question "你的架构问题"
npm run arch:mock -- --question "你的架构问题"
```

## 5. 详细设计文档输出

### 5.1 框架定位

用于在产品方案和技术架构明确之后，进一步输出可以指导开发、测试、联调和上线的详细设计文档。

流程：

```text
产品方案 / 技术架构
  -> 资深开发 A：数据模型、核心流程、事务和状态机设计
  -> 资深开发 B：服务拆分、中间件、接口、并发和运维设计
  -> 测试负责人：测试策略、用例矩阵、压测和故障演练质疑
  -> 两名资深开发修订
  -> 输出最终详细设计文档 final-detailed-design.md
```

### 5.2 角色配置

| 角色 | 建议模型/工具 | 职责 |
|---|---|---|
| 资深开发 A | Codex / GPT-5.5 / Claude Code | 数据表、索引、状态机、事务边界、核心算法、异常恢复 |
| 资深开发 B | Codex / GPT-5.5 / Claude Code | 服务模块、中间件结构、接口协议、并发性能、部署运维 |
| 测试负责人 | Codex / GPT-5.5 / Claude Code | 测试策略、用例矩阵、边界场景、压测、故障注入、验收门禁 |

### 5.3 阶段输出

```text
00-input.md
01-senior-dev-a-data-core-design.md
02-senior-dev-b-service-middleware-design.md
03-test-lead-test-design.md
04-developer-revision.md
05-final-detailed-design.md
```

最终详细设计文档应包含：文档范围、输入依据、目标与非目标、模块拆分、核心流程设计、状态机设计、数据库详细设计、索引与分库分表设计、Redis 详细设计、MQ 详细设计、其他中间件结构、接口详细设计、幂等并发与事务设计、异常重试与恢复设计、配置项设计、可观测性设计、测试设计、压测与容量评估、上线与回滚方案、风险清单与待确认问题。

### 5.4 推荐输入模板

```text
详细设计文档输出：
【产品方案】...
【技术架构】...
【重点】表结构、索引、Redis、MQ、接口、状态机、并发控制、测试方案
【希望输出】最终详细设计文档
```

## 6. 当前使用建议

- 如果还只有想法或业务目标，先用 `产品方案输出：`。
- 如果已经是系统设计问题，直接用 `架构评审：`。
- 如果已经有产品方案和架构方案，需要落到表、接口、中间件和测试，用 `详细设计文档输出：`。
- 如果某个角色失败，不要直接丢弃前面结果；保留已完成阶段，重试失败角色，必要时再基于已产物补齐后续阶段。
