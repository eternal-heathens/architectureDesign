# 两名资深开发 / 一名测试详细设计文档输出框架

## 1. 框架定位

该框架用于在产品方案和技术架构明确之后，进一步输出可以指导开发、测试、联调和上线的详细设计文档。

它关注的是“怎么实现到代码层和数据层”，输出应覆盖：

- 具体模块设计。
- 具体表结构和索引。
- Redis / MQ / 缓存 / 定时任务等中间件结构。
- 接口协议和错误码。
- 状态机、并发控制、幂等和事务边界。
- 测试用例、压测、故障演练和验收标准。

整体链路：

```text
产品方案 / 技术架构
  -> 输入过大时先压缩成结构化详细设计依据
  -> 资深开发 A：数据模型、核心流程、事务和状态机设计
  -> 资深开发 B：服务拆分、中间件、接口、并发和运维设计
  -> 测试负责人：测试策略、用例矩阵、压测和故障演练质疑
  -> 两名资深开发修订
  -> 输出最终详细设计文档 final-detailed-design.md
```

## 2. 角色配置

| 角色 | 建议模型/工具 | 职责 |
|---|---|---|
| 资深开发 A | Codex / GPT-5.5 / Claude Code | 数据表、索引、状态机、事务边界、核心算法、异常恢复 |
| 资深开发 B | Codex / GPT-5.5 / Claude Code | 服务模块、中间件结构、接口协议、并发性能、部署运维 |
| 测试负责人 | Codex / GPT-5.5 / Claude Code | 测试策略、用例矩阵、边界场景、压测、故障注入、验收门禁 |

## 3. 输入压缩规则

当产品方案、技术架构、历史设计、接口文档、表结构、日志、测试反馈或会议纪要过大时，必须先进行结构化压缩，再进入两名资深开发和一名测试的详细设计链路。

压缩目标不是简单缩短文字，而是保留研发落地必需信息：

- 产品目标、技术架构结论、模块边界和非目标。
- 核心业务流程、状态机、异常流和补偿逻辑。
- 数据表、字段、索引、唯一约束、数据规模和迁移要求。
- Redis、MQ、缓存、分布式锁、定时任务等中间件设计依据。
- 接口协议、错误码、鉴权、幂等、并发控制和事务边界。
- 性能目标、容量估算、部署约束、监控告警和运维要求。
- 测试范围、验收门禁、压测目标、故障演练和回归范围。
- 已确认结论、争议点、风险和未决问题。

压缩结果建议使用：

```markdown
## 输入压缩摘要
## 产品与架构依据
## 模块边界
## 核心流程与状态机
## 数据库与索引要点
## Redis / MQ / 中间件要点
## 接口与幂等并发要点
## 测试与压测要点
## 风险与未决问题
```

后续角色只能基于压缩摘要继续推演；如果发现摘要缺少关键实现依据，应先补问或回看原始材料中的相关片段。

## 4. 工作流

```text
0. 输入过大时：生成结构化输入压缩摘要
1. 输入产品方案和技术架构
2. 资深开发 A：核心实现与数据设计初稿
3. 资深开发 B：服务与中间件设计初稿，并质疑开发 A
4. 测试负责人：测试设计，并质疑两个开发方案
5. 资深开发 A/B：合并修订详细设计
6. 输出 final-detailed-design.md
```

## 6. 详细设计质量门禁

最终详细设计文档必须满足：

- 每张表都有 DDL、字段说明、主键、唯一索引、普通索引和索引用途。
- 每个 Redis key 都有 key 规则、value 类型、TTL、读写方、容量估算和清理策略。
- 每个 MQ topic 都有消息结构、生产者、消费者、消费语义、重试和死信策略。
- 每个核心接口都有请求、响应、错误码、幂等规则和鉴权要求。
- 每个状态流转都有触发条件、CAS 条件、失败处理和审计要求。
- 每个定时任务或调度器都有分片、锁、批量大小、失败重试和监控指标。
- 测试文档覆盖正常链路、异常链路、并发、幂等、故障恢复和压测。

## 7. 角色失败重试策略

- 单个角色超时或调用失败时，默认简单重试 1 次。
- 重试间隔默认 3 秒。
- 重试仍失败时，保留已完成阶段产物，并标记失败角色和失败原因。
- 配置错误类问题不反复重试，例如模型凭据缺失、provider 不匹配、命令不存在。
