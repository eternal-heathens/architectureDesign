# 三类方案输出触发词速查

## 1. 产品方案输出

推荐触发词：

```text
产品方案输出：
```

适用场景：

- 你有一个产品想法、业务问题、用户需求或功能设想。
- 希望先由 CEO 分析需求，再由产品总监输出产品化文档，最后由技术总监评估可行性和成本。
- 输入材料过大时，先压缩成结构化需求摘要，再进入产品方案输出链路。

推荐输入模板：

```text
产品方案输出：
【背景】...
【目标用户】...
【核心目标】...
【当前问题】...
【约束】...
【希望输出】最终产品方案
```

别名：

```text
需求方案输出：
产品需求评审：
```

## 2. 技术架构输出

推荐触发词：

```text
技术架构输出：
```

或者继续使用当前已验证的触发词：

```text
架构评审：
```

适用场景：

- 你已经有明确技术问题或系统设计问题。
- 希望跑技术总监、架构师、技术骨干三角色。
- 输出正式技术架构方案。
- 输入材料过大时，先压缩成结构化技术问题摘要，再进入架构评审链路。
- 遇到通用技术问题时，先调研成熟开源工具、框架或行业方案，再综合考虑自研、开源扩展或商业方案。

推荐输入模板：

```text
架构评审：
【背景】...
【目标】...
【约束】...
【关注点】高并发、可靠性、成本、扩展性、故障恢复、开源工具调研
```

别名：

```text
技术方案输出：
架构方案输出：
```

当前本机可执行命令：

```bash
npm run arch -- --question "你的架构问题"
```

## 3. 详细设计文档输出

推荐触发词：

```text
详细设计文档输出：
```

或者：

```text
详细设计：
```

适用场景：

- 产品方案和技术架构已经相对明确。
- 希望进一步细化到开发、测试、联调和上线可用的设计。
- 需要具体表结构、索引、Redis、MQ、接口、状态机、测试方案和压测方案。
- 输入材料过大时，先压缩成结构化详细设计依据，再进入详细设计输出链路。

推荐输入模板：

```text
详细设计文档输出：
【产品方案】...
【技术架构】...
【重点】表结构、索引、Redis、MQ、接口、状态机、并发控制、测试方案
【希望输出】最终详细设计文档
```

别名：

```text
详细设计：
研发详细设计：
```

## 4. 以后怎么避免重复确认

以后直接在 Codex 里用下面三个前缀即可：

```text
产品方案输出：
架构评审：
详细设计文档输出：
```

我会按前缀选择对应框架。如果你写的是别名，比如 `技术方案输出：`，我也按技术架构输出处理。

三类方案都默认执行输入压缩规则：当你贴的背景、历史方案、会议纪要、接口文档、日志或附件内容过大时，先生成 `输入压缩摘要`，保留目标、约束、已确认结论、关键数据、风险、争议和未决问题，再基于摘要继续输出方案。

技术架构输出默认遵循“开源优先、验证先行、综合决策”：遇到问题先看是否有成熟开源工具可以解决，再结合业务目标、稳定性、维护成本、安全合规、团队能力和长期演进决定采用、扩展、商业采购或自研。

如果你想强制走某个角色链路，可以在第一行写清楚：

```text
请按三角色技术架构输出：
```

或：

```text
请按两名资深开发和一名测试输出详细设计文档：
```
